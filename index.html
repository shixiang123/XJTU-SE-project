📌 Project Overview

XJTUhub is a campus learning assistant designed for Xi'an Jiaotong University students. The project contains two major parts: a WeChat mini-program frontend (XJTUhub-main) and a Node.js crawler-backed service (XJTU-API-main). The mini-program focuses on high-frequency student tasks (login, timetable, score, resources, attendance and DDL), while the backend focuses on stable data contracts and crawler-triggered synchronization from eHall-related systems.

Compared with a pure mock-data prototype, this project already includes a practical backend workflow: user login issues a local token, crawler tasks are triggered asynchronously, and mapped data is returned through unified API endpoints. At the same time, several modules preserve fallback data logic for robustness in network-unavailable or interface-unready cases.

📱 Frontend Feature Breakdown (XJTUhub-main)

Login & Auth UX

  • Two login entries: normal login and captcha login pages.
  • Credential validation, loading state, error toast, and post-login redirect handling.
  • Optional account caching (wx.setStorageSync("account")) for quick relogin.
  • After login success, frontend triggers backend crawler asynchronously.

Course Schedule Module

  • Weekly timetable with week switching (manual selection + swipe + step buttons).
  • Current-week auto computation from configured term start date.
  • Course detail popup with week/day/section/location/teacher/credit/category/method.
  • Course color assignment with cache, versioning, and light/dark mode adaptation.

Score Module

  • Term-level score list viewing with term selector.
  • Raw score detail mapping (normal/midterm/final/skill/composite).
  • Graceful value formatting (missing values shown as --).
  • Dual-cache strategy for valid score list and raw score list.

Resource Plaza Module

  • Category filtering (courseware, notes, exams), keyword search and pagination.
  • Pull-down refresh + reach-bottom loading behavior.
  • 8-second timeout fallback to local demo resources to keep UI always available.
  • Resource normalization for multiple backend payload shapes.

Assistant Modules (Attendance / DDL / Function / Mine)

  • Attendance page with risk-tier demo data (normal/warning/danger) and local cache.
  • DDL list with completion toggle and one-click demo reset.
  • Function hub for quick navigation to score, attendance and DDL pages.
  • Profile page with auth-state display, shortcut menu and logout operation.

UI Engineering Details

  • Custom TabBar with active-route sync and theme sync.
  • Page enter animation pattern reused across major pages.
  • Unified request wrapper for token injection, error code handling and login redirection.
  • Theme mode support across page components (light mode baseline + dark mode adaptation logic).

⚙️ Backend Capability Breakdown (XJTU-API-main)

Auth and Compatibility Layer

  • Returns local token in format xjtu-<stuId>-<timestamp>-<random>.
  • Keeps /login-init, /login-code, /login-verify for compatibility with mini-program routes.
  • Uses request-header token parsing for scoped data retrieval by student id.
  • Maintains unified response structure: { code, msg, data }.

Crawler Task Orchestration

  • Supports target-based crawling: all / course / score.
  • Background process spawning with timeout protection and log tail capture.
  • Runtime state includes running flag, active target, timestamps, last error and summary.
  • Crawler output is synchronized and mapped before API exposure.

Mapped Data Endpoints

  • /courses returns normalized course entries from crawler-mapped data.
  • /scores and /raw-scores return term-grouped score datasets.
  • Returns empty arrays when mapped data is unavailable, preventing frontend crashes.
  • Status endpoint provides summary counts for monitoring and UI hinting.

Resource Service

  • File-list API with pagination, category and keyword filtering.
  • Upload/delete support with ownership check via token-derived uploader id.
  • File metadata persistence (.meta.json) and deterministic resource id generation.
  • Material URL generation under /materials/* path.

🔄 End-to-End Data Flow

  1. User submits account/password from mini-program login page and receives local token.
  2. Frontend stores token and immediately triggers crawler task in background.
  3. Crawler script logs into eHall workflow and exports raw data artifacts.
  4. Backend maps raw data into frontend contract objects (course/score/raw-score).
  5. Mini-program pages request mapped endpoints and cache results locally for quick display.
  6. If a module endpoint is delayed/unavailable, frontend fallback logic preserves usability.
